Standard Coverages

Each type of dwelling has a different set of Standard Coverages (StdCov structure) that are rated with the dwelling. FORTUS™ will create and attach the necessary Standard Coverage structures based on the type of dwelling, if they do not exist in the input XML. Therefore, the input XML does not need to contain any Standard Coverages.

Coverage Results

Coverage results require some interpretation. The following illustrates how to interpret coverage results:

  • If <Premium> = 0, then the coverage is Included.
  • If a coverage was requested in the input, but did not appear in the output, then it is Not Available.
  • If a coverage contains a <Message> tag(s), then the Message should be examined to determine if the coverage was available or not. In most circumstances, the existence of a <Message> tag(s) indicates that the coverage was Not Available.

ID Fields

Each structure has an ID field. This field is described as a 'Unique ID string'. The value must only be unique for immediate siblings of it's containing parent. For example, two claims can be defined. These claim structures must contain unique ID values, say 'C1' and 'C2'.

Multiple Forms

Some carriers offer multiple forms for a coverage type. For example, a carrier might offer two different forms for All Risks/All Risks. FORTUS™ can now quote both forms simultaneously. For backwards compatibility, multiple forms quoting is turned OFF by default. To quote multiple forms, do one of the following:

  • Go to Settings in Fortus Config and check "Habitation - Multiple Forms"
  • Include the following in the input XML as a child node of <FortusTransaction>:
    <RegistryOverrides>
    <HabMultipleForms>1</HabMultipleForms>
    </RegistryOverrides>

If not quoting multiple forms, the form quoted will be the default form. If quoting multiple forms, additional forms will manifest as additional <QuoteInfo> tags. These additional tags are identified with an AdditionalForm attribute with a value of "TRUE".

Lead Detail

When talking about leads and the parties involved, we can use the following terms:

  • Submitter: The web site or system submitting the lead to the lead recipient.
  • Recipient: The brokerage/system targeted to receive the lead.
  • Client: The consumer who is the topic of the lead.
  • Referrer: The web site/system which referred the client to the submitter.

Example

Below is an example of a fulsome LeadDetail section, with interpretation:

  • Origin tag helps convey that MyMaxInsurance.ca is submitting the lead.
  • This lead is providing some detail about itself:
    • This lead/client is identified as "X-1234567" within MyMaxInsurance.ca's systems.
    • This lead is reprenting a quote where company "GA" yielded a premium of "1234".
    • The consumer arrived at MyMaxInsurance.ca from Facebook.
    • MyMaxInsurance.ca describes the lead as being of type "Partial".
  • Contact preferenced details are provided for the lead recipient, to reach out and communicate with the client:
    • The client would prefer to be contacted by phone, in the afternoon, in English.
  • MyMaxInsurance.ca provides some detail of the lead recipient, from it's point of view:
    • The lead recipient has an account code of "ACME" within MyMaxInsurance.ca's system.
    • The full name of that account, again within MyMaxInsurance.ca's system, is "Acme Insurance Ltd.".
